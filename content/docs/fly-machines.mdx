# Fly Machines Provisioner

Set `CONTAINER_BACKEND=flyio` and every `POST /workspaces` call provisions a Fly Machine inside your Fly app instead of a local Docker container. Workspaces spin up in 15–30 seconds, idle machines scale to zero, and you can place workers in any of Fly's 35+ regions by setting a single env var.

<Callout type="warn">
  **Current scope (PR #501):** Workspace images must already be published to GHCR before provisioning. The `delete` and `restart` platform endpoints are not yet fully wired to the Fly provisioner — use `flyctl machine stop/destroy` for teardown until a follow-up PR lands.
</Callout>

---

## When to use this backend

| Scenario | Recommended backend |
|---|---|
| Local dev / self-hosted | `docker` (default — no external dependencies) |
| Indie dev / small team on Fly.io | `flyio` — pay-per-use, scale to zero, 35+ regions |
| SaaS / multi-tenant product | `controlplane` — Fly credentials held centrally, never on the tenant |

If you are building a multi-tenant SaaS product, see the [Quickstart](/docs/quickstart) backend selection guide. The `controlplane` backend removes the Fly token from the tenant layer entirely — it's a structural credential isolation choice, not a config tweak.

## Prerequisites

- A Molecule AI platform instance
- A [Fly.io](https://fly.io) account with a Fly app created for workspace machines
- `flyctl` installed locally (`curl -L https://fly.io/install.sh | sh`)
- `curl` + `jq`

## Configuration

Set the following environment variables on your platform process (add to `.env`, your deployment config, or `flyctl secrets set` on the platform app):

| Variable | Required | Default | Description |
|---|---|---|---|
| `CONTAINER_BACKEND` | Yes | — | Set to `flyio` to activate the Fly provisioner |
| `FLY_API_TOKEN` | Yes | — | Fly deploy token — run `flyctl tokens create deploy` |
| `FLY_WORKSPACE_APP` | Yes | — | Name of the Fly app that will host workspace machines |
| `FLY_REGION` | No | `ord` | Default region for new machines |

```bash
export CONTAINER_BACKEND=flyio
export FLY_API_TOKEN=<your-fly-deploy-token>
export FLY_WORKSPACE_APP=my-molecule-workspaces
export FLY_REGION=ord  # optional — defaults to ord
```

After setting these, restart the platform so it picks up `CONTAINER_BACKEND=flyio`.

## Resource tiers

Workspaces are sized by tier. Pass `"tier": 2`, `3`, or `4` in `POST /workspaces`. The Fly provisioner applies limits automatically — no manual machine sizing required.

| Tier | RAM | vCPUs | Recommended for |
|---|---|---|---|
| T2 | 512 MB | 1 | Light workers, eval agents |
| T3 | 2 GB | 2 | General-purpose orchestrators |
| T4 | 4 GB | 4 | Heavy inference, long-context tasks |

Runtime images are resolved from GHCR automatically: `hermes` → `ghcr.io/molecule-ai/workspace-hermes:latest`, etc.

## Quickstart

```bash
# 1. Verify the platform is using the Fly provisioner
curl -s http://localhost:8080/healthz | jq .

# 2. Create a workspace — the platform provisions it as a Fly Machine
WS=$(curl -s -X POST http://localhost:8080/workspaces \
  -H "Content-Type: application/json" \
  -d '{
    "name": "fly-worker",
    "role": "Fly-provisioned inference worker",
    "runtime": "hermes",
    "tier": 2
  }' | jq -r '.id')
echo "Workspace ID: $WS"

# 3. Watch the Fly Machine appear (15–30s)
flyctl machines list --app $FLY_WORKSPACE_APP

# 4. Poll until the workspace is ready
until curl -s http://localhost:8080/workspaces/$WS \
    | jq -r '.status' | grep -q ready; do
  echo "Waiting..."; sleep 5
done

# 5. Smoke test — confirm A2A is routing through the Fly Machine
curl -s -X POST http://localhost:8080/workspaces/$WS/a2a \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "id": "smoke-1",
    "method": "message/send",
    "params": {
      "message": {
        "role": "user",
        "parts": [{"kind": "text", "text": "What region are you running in?"}]
      }
    }
  }' | jq '.result.parts[0].text'
# Expected: "ord" (or whichever FLY_REGION you set)

# 6. Inspect the Fly Machine
flyctl machines show --app $FLY_WORKSPACE_APP

# 7. Teardown — use flyctl directly (see scope note above)
flyctl machines destroy --app $FLY_WORKSPACE_APP --force
```

## Regions

Set `FLY_REGION` to any [Fly region code](https://fly.io/docs/reference/regions/) in your platform env to control where new machines provision by default. Fly supports 35+ regions — place workers close to your users or your data sources with no infrastructure code changes.

Because the Fly provisioner implements the same `Provisioner` interface as the Docker backend, the rest of the platform is entirely unchanged: same REST API, same A2A protocol, same workspace management UI. Switching backends requires no changes to agent code, tool definitions, or orchestration logic.

---

## How provisioning works

When a workspace is created, the Fly provisioner:

1. Selects the GHCR image for the requested runtime
2. Creates a Fly Machine in `FLY_WORKSPACE_APP` with tier-based resource limits applied
3. Injects workspace secrets, `PLATFORM_URL`, and the workspace ID as machine env vars
4. Issues a boot-time auth token so the agent registers with the platform on first start

The workspace agent boots, calls back to the platform registration URL, and transitions to `ready` status — the same boot sequence as Docker-provisioned workspaces.

## Troubleshooting

### Workspace stuck in `provisioning` status

Run `flyctl machines list --app $FLY_WORKSPACE_APP`. If the machine is in a `failed` state, view logs with `flyctl logs --app $FLY_WORKSPACE_APP`. Common causes:

- GHCR image not yet published for the requested runtime
- Invalid or expired `FLY_API_TOKEN`
- `FLY_WORKSPACE_APP` does not exist in your Fly account

### `FLY_API_TOKEN` rejected

Run `flyctl tokens create deploy` to issue a new deploy token. The token needs `Machine > write` and `Apps > read` scopes on the target app.

### Machine starts but workspace never reaches `ready`

The agent registered before the platform was reachable, or `PLATFORM_URL` was misconfigured. Check platform logs for registration attempts. Ensure the platform is publicly accessible from Fly's network, or that the platform and workspace app are on the same [Fly private network](https://fly.io/docs/networking/private-networking/).
